      10                 :            : /*
      11                 :            :  * This module exists to support sorting via multiple merge passes (see
      12                 :            :  * tuplesort.c).  Merging is an ideal algorithm for tape devices, but if
      13                 :            :  * we implement it on disk by creating a separate file for each "tape",
      14                 :            :  * there is an annoying problem: the peak space usage is at least twice
      15                 :            :  * the volume of actual data to be sorted.      (This must be so because each
      16                 :            :  * datum will appear in both the input and output tapes of the final
      17                 :            :  * merge pass.  For seven-tape polyphase merge, which is otherwise a
      18                 :            :  * pretty good algorithm, peak usage is more like 4x actual data volume.)
      19                 :            :  *
      20                 :            :  * We can work around this problem by recognizing that any one tape
      21                 :            :  * dataset (with the possible exception of the final output) is written
      22                 :            :  * and read exactly once in a perfectly sequential manner.      Therefore,
      23                 :            :  * a datum once read will not be required again, and we can recycle its
      24                 :            :  * space for use by the new tape dataset(s) being generated.  In this way,
      25                 :            :  * the total space usage is essentially just the actual data volume, plus
      26                 :            :  * insignificant bookkeeping and start/stop overhead.
      27                 :            :  *
      28                 :            :  * Few OSes allow arbitrary parts of a file to be released back to the OS,
      29                 :            :  * so we have to implement this space-recycling ourselves within a single
      30                 :            :  * logical file.  logtape.c exists to perform this bookkeeping and provide
      31                 :            :  * the illusion of N independent tape devices to tuplesort.c.  Note that
      32                 :            :  * logtape.c itself depends on buffile.c to provide a "logical file" of
      33                 :            :  * larger size than the underlying OS may support.
      34                 :            :  *
      35                 :            :  * For simplicity, we allocate and release space in the underlying file
      36                 :            :  * in BLCKSZ-size blocks.  Space allocation boils down to keeping track
      37                 :            :  * of which blocks in the underlying file belong to which logical tape,
      38                 :            :  * plus any blocks that are free (recycled and not yet reused).
      39                 :            :  * The blocks in each logical tape are remembered using a method borrowed
      40                 :            :  * from the Unix HFS filesystem: we store data block numbers in an
      41                 :            :  * "indirect block".  If an indirect block fills up, we write it out to
      42                 :            :  * the underlying file and remember its location in a second-level indirect
      43                 :            :  * block.  In the same way second-level blocks are remembered in third-
      44                 :            :  * level blocks, and so on if necessary (of course we're talking huge
      45                 :            :  * amounts of data here).  The topmost indirect block of a given logical
      46                 :            :  * tape is never actually written out to the physical file, but all lower-
      47                 :            :  * level indirect blocks will be.
      48                 :            :  *
      49                 :            :  * The initial write pass is guaranteed to fill the underlying file
      50                 :            :  * perfectly sequentially, no matter how data is divided into logical tapes.
      51                 :            :  * Once we begin merge passes, the access pattern becomes considerably
      52                 :            :  * less predictable --- but the seeking involved should be comparable to
      53                 :            :  * what would happen if we kept each logical tape in a separate file,
      54                 :            :  * so there's no serious performance penalty paid to obtain the space
      55                 :            :  * savings of recycling.  We try to localize the write accesses by always
      56                 :            :  * writing to the lowest-numbered free block when we have a choice; it's
      57                 :            :  * not clear this helps much, but it can't hurt.  (XXX perhaps a LIFO
      58                 :            :  * policy for free blocks would be better?)

      60                 :            :  * To support the above policy of writing to the lowest free block,
      61                 :            :  * ltsGetFreeBlock sorts the list of free block numbers into decreasing
      62                 :            :  * order each time it is asked for a block and the list isn't currently

      64                 :            :  * of releasing many blocks followed by re-using many blocks, due to
