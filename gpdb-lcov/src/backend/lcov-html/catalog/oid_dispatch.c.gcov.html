       3                 :            :  * oid_dispatch.c
       4                 :            :  *              Functions to ensure that QD and QEs use same OIDs for catalog objects.
       5                 :            :  *
       6                 :            :  *
       7                 :            :  * In Greenplum, it's important that most objects, like relations,
       8                 :            :  * functions, operators, have the same OIDs in the master and all QE nodes.
       9                 :            :  * Otherwise query plans generated in the master will not work on the QE
      10                 :            :  * nodes, because they use the master's OIDs to refer to objects.
      11                 :            :  *
      12                 :            :  * Whenever a CREATE statement, or any other command that creates new
      13                 :            :  * objects, is dispatched, the master also needs to tell the QE servers which
      14                 :            :  * OIDs to use for the new objects. Before GPDB 5.0, that was done by
      15                 :            :  * modifying all the structs representing DDL statements, like DefineStmt,
      16                 :            :  * CreateOpClassStmt, and so forth, by adding a new OID field to them.
      17                 :            :  * However, that was annoying when merging with the upstream, because it
      18                 :            :  * required scattered changes to all the structs, and the accompanying
      19                 :            :  * routines to copy and serialize them. Moreover, for more complicated object
      20                 :            :  * types, like a table, a single OID was not enough, as CREATE TABLE not only
      21                 :            :  * creates the entry in pg_class, it also creates a composite type, an array
      22                 :            :  * type for the composite type, and possibly the same for the associated
      23                 :            :  * toast table.
      24                 :            :  *
      25                 :            :  * Starting with GPDB 5.0, we take a different tact. Whenever an OID is
      26                 :            :  * assigned for a system table in the QD node, the OID is recorded in private
      27                 :            :  * memory, in the 'dispatch_oids' list, along with the key for that object.
      28                 :            :  * AddDispatchOidFromTuple() function does that, and it's called from
      29                 :            :  * heap_insert(). For example, when a new type is created, the OID of the new
      30                 :            :  * type, and the namespace and the type name of the new type, are recorded in
      31                 :            :  * the 'dispatch_oids' list.  When the command is dispatched to the QE
      32                 :            :  * servers, all the recorded OIDs are included in the dispatched request, and
      33                 :            :  * the QE processes in turn stash the list into backend-private memory. This
      34                 :            :  * is the 'preassigned_oids' list.
      35                 :            :  *
      36                 :            :  * In a QE node, when we are about to create a new object, and assign an OID
      37                 :            :  * for it, we look into the 'preassigned_oids' list to see if we had received
      38                 :            :  * an OID to use for the named object from the master. Under normal
      39                 :            :  * circumstances, we should have a pre-assigned OID for all objects created
      40                 :            :  * in QEs, and the GetPreassigned*() functions will throw an error if we
      41                 :            :  * don't.
      42                 :            :  *
      43                 :            :  * All in all, this provides a generic mechanism for DDL commands, to record
      44                 :            :  * OIDs that are assigned for new objects in the master, transfer them to QE
      45                 :            :  * nodes when the DDL command is dispatched, and for the QE nodes to use the
      46                 :            :  * same, pre-assigned, OIDs for the objects.
      47                 :            :  *
      48                 :            :  * This same mechanism can be used to preserve OIDs during pg_upgrade. In
      49                 :            :  * PostgreSQL, pg_upgrade only needs to preserve the OIDs of a few objects,
      50                 :            :  * like types, but in GPDB we need to preserve most OIDs, because they need
      51                 :            :  * to be kept in sync between the nodes. (Strictly speaking, we only need to
      52                 :            :  * ensure that all the nodes use the same OIDs in the upgraded clusters, but
      53                 :            :  * they wouldn't need to be the same as before upgrade. However, the most
      54                 :            :  * straightforward way to achieve that is to use the same OIDs as before
      55                 :            :  * upgrade.)
      56                 :            :  *
      57                 :            :  * pg_upgrade has its own mechanism to record the OIDs from the old cluster
      58                 :            :  * but when restoring the schema in the new cluster, it uses the same
